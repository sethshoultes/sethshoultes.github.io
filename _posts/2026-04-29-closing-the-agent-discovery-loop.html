Visit https://smartwebutah.com/.well-known/mcp.json from any HTTP client — no credentials, no headers, no session. The response, in full:

{
  "name": "Smart Website Solutions - Utah",
  "description": "Search Engine Optimization, Website Security, and Business Consulting in Southern Utah",
  "protocol": "mcp",
  "version": "2025-06-18",
  "endpoints": { "public": "https://smartwebutah.com/wp-json/mp-mcp/v1/public-mcp" },
  "auth_required": false,
  "transport": "http"
}

That JSON points to a JSON-RPC 2.0 endpoint. POST {"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}} to the listed URL and an AI client can enumerate the site's membership plans, fetch pricing, and retrieve signup URLs. No credentials. No human in the loop. This is the first time the agent discovery loop has fully closed for a real commerce surface.

The loop and what it takes to close it

The loop has three steps. The agent visits a URL, or is given one. The agent fetches /.well-known/mcp.json to learn what's available there. The agent queries the discovered endpoint and gets back structured catalog data with no authentication. Skip any step and the loop is open. All three have to work, in sequence, for the loop to close.

Most implementations stop at step two, or between two and three. They build an MCP endpoint that requires a bearer token, wire up the token exchange, and call it done. This is the right architecture for a site owner who wants to query their own membership data from Claude Desktop — authenticated session, stored token, owner running their own queries against their own site. It is the wrong architecture for the prospective customer who asks the same Claude Desktop "what does this site offer and how much does it cost?"

The prospect has no token. The site owner cannot share a token with every prospect — that is not a token, that is a public credential, which is something else entirely. Even if the prospect's AI client somehow knew the URL and knew the endpoint, it could not authenticate. There was no zero-credential path. The loop was open at step three.

The asymmetry

A human visitor arriving at a membership site has resources an AI client does not. Cookies from a previous visit. A browser that renders the pricing page as a readable surface. The ability to click "See plans" and scroll. The capacity to make inferences from layout — this box costs more than that box, this tier includes these features. None of that is available to an agent arriving at the same URL. The agent has HTTP. The agent has headers. If the response to an unauthenticated request is a 401, or a WordPress login redirect, or a marketing page with no structured data, the agent has nothing it can reason from.

The asymmetry is not a minor inconvenience. It is a wall. And the wall does not announce itself. An agent that cannot authenticate against an MCP endpoint does not receive a helpful explanation of what it would need. It receives a 401. The loop is open and the agent has no way to know whether the loop was ever meant to close.

The public MCP layer is the same fix applied to a different surface. The site had an ear — the authenticated endpoint — that only the owner could speak into. The prospective customer needed a different ear, with a different contract. The public layer is that contract.

Three files

Closing the loop required three files in MemberPress AI Foundation. Each one does one thing. All three are required.

PublicServer.php is a JSON-RPC 2.0 endpoint registered with WordPress's REST API at /wp-json/mp-mcp/v1/public-mcp, with permission_callback returning true — no authentication required. It accepts the standard MCP methods: tools/list, tools/call, initialize. Rate limiting sits at 60 requests per minute per IP address, enforced in the request handler before any database query runs. The endpoint sends Cache-Control: public, max-age=300 headers so high-traffic catalog queries don't hammer the database on every request. The public endpoint is the surface. Without it, the discovery file points nowhere actionable.

MembershipFormatter::public_detail() is the formatter method that shapes what the endpoint returns. This is not a minor implementation detail. The formatter is the wall between the public surface and the site's internal data. It strips member counts, access rules, custom meta, and internal notes. It adds the public signup URL — the one a prospective customer would need to convert. A misconfigured tool that calls the wrong method, or a future developer who adds a field to the wrong response, cannot leak admin data through this endpoint, because the formatter method does not have access to that data. It returns exactly what a prospective customer needs to know: name, price, billing interval, features, URL. The wall is in the formatter, not in the permissions check.

Auth::handle_well_known() serves the discovery file. When the admin has enabled the public MCP toggle, the method responds to GET /.well-known/mcp.json with the JSON that SmartWebUtah.com serves. When the toggle is off, the method returns 404. The discovery file's existence is itself opt-in. An agent fetching /.well-known/mcp.json on a site where the toggle is off will find nothing — not a JSON blob announcing "authentication required," but a 404, as if the file does not exist. Which, from the agent's perspective, it doesn't.

The default is off

Public surfaces should not be implicit. The plugin ships with the public MCP toggle disabled. The site owner opts in via a checkbox in the AI Settings admin page. The discovery file does not exist until they do.

This is not an abundance of caution. It is the only defensible default for a surface that exposes structured catalog data with no authentication. A site owner who enables it has made an explicit decision. A site owner who installs the plugin and does nothing has made no such decision, and the plugin respects that. The wall is not just in the formatter — it starts at the toggle.

There is a second layer of opt-out below the site-level toggle. Each membership and course post-edit screen has a meta box where the site owner can flag specific products as excluded from the public catalog. A site that primarily sells B2B memberships with custom pricing, or that has internal tiers not intended for direct signup, can leave the toggle on for the public-facing products and exclude the rest. The formatter checks the exclusion flag before building a product's response.

The discovery file closes the loop

The public endpoint alone — even a well-designed one — does not close the loop. An agent that does not know the endpoint exists cannot query it. An agent that knows the site's domain but not its MCP URL has to guess at URL patterns, which is not a reliable mechanism and is not how the MCP specification anticipates discovery working.

The /.well-known/mcp.json convention solves the discovery problem the same way /.well-known/acme-challenge solves certificate issuance or /.well-known/openid-configuration solves OAuth discovery: a predictable path at a predictable location that any client can check without prior knowledge of the site's internal structure. An agent that knows a domain can check /.well-known/mcp.json. If the file is there, the agent knows the endpoint URL, the protocol version, the transport type, and whether authentication is required. If the file is not there — because the toggle is off, or because the plugin is not installed — the agent receives a 404 and moves on.

MemberPress AI Foundation v1.1.0 shipped the public endpoint; v1.2.0 shipped the discovery file. The endpoint without the file was a partial loop — useful for integrators who already knew the URL, useless for agents operating without prior knowledge of the site. The discovery file is what changes an endpoint into a discoverable service. That distinction is the difference between a loop that is mostly closed and one that is actually closed.

What the loop closing means

The loop is closed for SmartWebUtah.com. An AI client given that domain can fetch the discovery file, find the endpoint, call tools/list, and enumerate the site's offerings without credentials and without a human in the loop. The membership plans, prices, billing intervals, and signup URLs come back as structured JSON that an LLM can reason from directly.

The Mouth Was Not an Ear described a site that had listeners but no ear the client could reach: the avatar SDK had three commands for output and none for input. The pipes ran one way. This is the inversion. The site had an authenticated MCP layer the owner could query and nothing the prospective customer's agent could find. The public layer is the ear the site was missing. The loop is now symmetrical: the agent can find the surface, and the surface is listening.

The engineering pattern

The agent discovery loop is an engineering problem, not a marketing problem. The people who talk about "AI search," "AI shopping," and "agentic commerce" as inevitable features of the near-term web are right that the direction of travel is clear. They are wrong to speak in future tense. The surface does not materialize on its own. The surface has to be built. And it is a specific three-file design with a default-off rule, not a trajectory or a tendency.

Anyone building commerce-adjacent software in 2026 who does not ship this triple — discoverable endpoint, public formatter, well-known discovery file — is building a surface that the site owner can use and the prospective customer's agent cannot find. The authenticated layer alone is a half-built ear. The prospective customer is standing at the door, and the site is not listening.

The loop closes or it doesn't. There is no middle state where it almost closes and the agent improvises the rest.
